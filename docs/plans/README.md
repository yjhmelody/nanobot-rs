# RFC 索引

本目录用于维护 `nanobot-rs` 的中长期改进 RFC。后续讨论、实施、调整和状态同步都应优先引用 RFC 编号，而不是模糊描述文件名。

## 状态约定

### 文档状态

- `Draft`：草稿，内容尚未稳定
- `Proposed`：已形成可讨论方案
- `Accepted`：方案已确认，可进入实施
- `Superseded`：已被新的 RFC 替代
- `Archived`：保留归档，不再推进

### 实施状态

- `Not Started`：未开始实施
- `In Progress`：正在实施
- `Blocked`：被依赖或外部条件阻塞
- `Partially Implemented`：已落地一部分
- `Completed`：已全部完成

## 使用规则

- RFC 编号**只表示方案提出的时间顺序**，不表示优先级、依赖层级或实施先后
- 当前目录中的 RFC 编号应始终保持**严格升序、连续递增、无跳号**
- 新增计划时按当前最大编号继续递增，例如 `RFC-0006`
- 更新方案时直接修改对应 RFC，而不是新建同主题散落文档
- 如果后续发现编号出现空洞或乱序，应统一重排为连续编号，并同步更新索引与交叉引用
- 实施过程中至少同步更新：
  - 文档头部的 `实施状态`
  - `实施阶段` 表格中的阶段状态
  - 必要时在索引表同步整体状态
- 需要推进时可以直接说：
  - “开始实施 `RFC-0002`”
  - “补充 `RFC-0003` 的审批规则”
  - “把 `RFC-0004` 标记为 `In Progress`”

## 建议实施顺序

以下顺序仅表示建议推进顺序，**不等同于 RFC 编号含义**。

1. `RFC-0001`：确认整体方向和优先级
2. `RFC-0002`：先补齐 CLI 工作流与状态模型
3. `RFC-0003`：再补工具调度和审批闭环
4. `RFC-0004`：升级记忆和子代理管理
5. `RFC-0005`：最后补产品化、诊断和发布链路
