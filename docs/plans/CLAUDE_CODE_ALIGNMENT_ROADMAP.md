# nanobot-rs 对标 claude-code 改进路线图

## 目标

参考 `claude-code` 在工作流、交互、安全、子代理、记忆和产品化上的亮点，为 `nanobot-rs` 制定一套可以分阶段实施的改进路线。重点不是复制完整 UI，而是把高价值能力抽象出来，按成本和收益逐步落地。

## 对标原则

- 优先借鉴“能力与机制”，不机械复制具体终端 UI
- 优先补齐高频工作流中的断点，而不是一次性追求完整产品形态
- 所有改进都要尽量复用当前 Trait-first 架构
- 阶段设计必须可独立交付、可验证、可回滚

## 参考仓库亮点摘要

`claude-code` 的亮点主要集中在以下几个方面：

- 明确的工作流状态：plan mode、review、resume、background task
- 清晰的进度反馈：intent、tool 分组、任务列表、状态面板
- 工具执行更安全：diff-first、权限规则、并发读/串行写
- 记忆更主动：session memory、durable memory、magic docs
- 子代理更产品化：后台运行、状态可见、隔离边界明确
- 产品成熟度更高：trust 流程、managed settings、MCP 审批、发布分发更完整

## 当前主要不足

### 1. 工作流能力偏底层

- CLI 只有基础命令，缺少 `plan`、`resume`、`sessions`、`tasks`、`review` 等工作流入口
- `status` 信息过薄，无法反映 provider、MCP、cron、session、subagent 的真实状态
- 交互式 CLI 仍是基础输入循环，缺少阶段感、任务感、进度聚合

### 2. 工具编排和安全仍偏“执行器”而非“产品能力”

- 多工具计划当前只执行第一个工具调用
- 缺少读写分级、审批门、diff 预览、权限规则
- 文件和 shell 工具以直接执行为主，风险控制粒度不够

### 3. 记忆与子代理能力还停留在基础版

- 记忆检索没有 query-aware 检索和分层抽取
- 长期记忆写入方式较粗
- 子代理缺少状态、预算、结果归集和生命周期管理

### 4. 产品化能力不足

- 可观测性仍以日志为主
- 安装、发布、命名和分发链路不完整
- 缺少一套面向用户的“自检、诊断、能力说明”入口

## 建议分阶段实施

## 阶段 0：快速修补当前明显断点

### 目标

先修掉会直接影响后续演进和用户信任的问题。

### 范围

- 统一二进制命名、文档命名、release workflow
- 修复 CI / 文档 / 模板中的明显漂移
- 增强 `status` 输出，至少能看到 provider、workspace、session 存储目录、已启用渠道
- 为后续 workflow 能力预留统一事件类型和状态模型

### 产出

- 命名一致的 CLI 和发布产物
- 基础可诊断的 `status` / `doctor`
- 后续阶段可复用的状态数据结构

## 阶段 1：CLI 工作流升级

### 目标

让 `nanobot-rs` 从“能跑的 agent runtime”升级到“可持续使用的终端助手”。

### 范围

- 新增 `plan`、`sessions`、`resume`、`tasks` 命令组
- 引入轻量 plan mode
- 引入任务/进度聚合展示
- 为交互式 CLI 增加历史、补全、快捷命令和更清晰的系统消息

### 预期收益

- 用户能明确知道当前处于“规划、执行、审阅、恢复”哪一个阶段
- 多轮任务更容易恢复和继续
- 后续 review/safety 功能有稳定入口

## 阶段 2：工具编排与安全升级

### 目标

让工具系统从“逐条调用”升级为“可调度、可审阅、可限制”的执行框架。

### 范围

- 支持多工具计划执行
- 引入并发读 / 串行写的执行策略
- 文件写入和 shell 执行增加审批与策略层
- 加入权限规则与信任级别

### 预期收益

- 明显减少 agent 反复往返
- 降低误写文件、误执行命令的风险
- 为后续 review/QA 自动化打基础

## 阶段 3：记忆与子代理升级

### 目标

让 `nanobot-rs` 从“每轮靠上下文堆叠”提升到“有项目长期记忆、可拆分任务、可追踪后台工作”的状态。

### 范围

- 分层记忆：session / project / manual
- query-aware 记忆检索与提炼
- 子代理任务状态、结果归集、预算限制
- 后台任务列表和恢复能力

### 预期收益

- 长任务更稳定
- 上下文压力更低
- 复杂任务可拆分并行执行

## 阶段 4：产品化与运维能力补齐

### 目标

补齐可观测性、分发和运维能力，让项目更像一款可交付产品。

### 范围

- 事件、指标、trace、health
- `doctor`、`status --verbose`、`tasks inspect`
- 多平台发布、校验和、安装脚本、包管理分发
- 配置参考文档和运维手册

### 预期收益

- 降低排障成本
- 提升首次安装成功率
- 便于社区使用和反馈

## 优先级建议

### P0

- 阶段 0 全部
- 阶段 1 中的 `sessions` / `resume` / 增强 `status`

### P1

- 阶段 1 中的 plan mode
- 阶段 2 中的读写分级、diff 审批、工具批执行

### P2

- 阶段 3 中的分层记忆、子代理任务面板
- 阶段 4 中的可观测性基础设施

### P3

- 更丰富的终端 UI
- 更强的 review / QA 自动化
- 更完整的团队级策略与信任体系

## 关键设计约束

- 不引入和现有 Trait-first 架构冲突的重型耦合层
- 不要求一开始就做完整 TUI，先把 workflow 和状态模型打通
- 任何默认更严格的安全策略都要提供清晰的用户反馈，避免“静默失败”
- 新能力优先通过 `nanobot-config` 暴露，并确保 CLI / gateway 共用

## 成功标准

- 用户可以用命令明确完成规划、执行、恢复、查看状态四类核心工作流
- 至少一类写操作具备 diff 审批或显式确认机制
- 至少一类子代理任务可后台运行并被查询状态
- `status` / `doctor` 能覆盖常见配置与运行故障
- 发布产物、文档和二进制命名保持一致
