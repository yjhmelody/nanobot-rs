## 摘要

参考 `claude-code` 在工作流、交互、安全、子代理、记忆和产品化上的亮点，为 `nanobot-rs` 制定一套分阶段实施路线。目标不是复制完整终端 UI，而是抽取高价值能力，以最小耦合方式嫁接到当前 Trait-first 架构中。

## 当前主要不足

### 工作流能力偏底层

- CLI 缺少 `plan`、`resume`、`sessions`、`tasks`、`review` 等工作流入口
- `status` 信息过薄，无法反映 provider、MCP、cron、session、subagent 的真实状态
- 交互式 CLI 缺少阶段感、任务感和进度聚合

### 工具编排与安全仍偏“执行器”

- 多工具计划当前只执行第一个工具调用
- 缺少读写分级、审批门、diff 预览、权限规则
- 文件和 shell 工具以直接执行为主，风险控制粒度不够

### 记忆与子代理仍是基础版

- 记忆检索没有 query-aware 检索和分层抽取
- 长期记忆写入方式较粗
- 子代理缺少状态、预算、结果归集和生命周期管理

### 产品化能力不足

- 可观测性仍以日志为主
- 安装、发布、命名和分发链路不完整
- 缺少一套面向用户的诊断与自检入口

## 对标结论

`claude-code` 最值得借鉴的不是复杂 UI，而是这些能力层：

- 明确的工作流状态：plan mode、resume、background task、review
- 清晰的系统反馈：intent、任务列表、状态面板、分组工具反馈
- 更安全的执行机制：并发读 / 串行写、diff-first、权限规则、信任模式
- 更主动的知识积累：session memory、durable memory、受限后台 worker
- 更成熟的产品化机制：trust、审批、诊断、发布与分发

## 方案总览

### 方向一：CLI 工作流升级

通过 `sessions`、`resume`、`tasks`、`plan`、`doctor` 等命令，建立“规划、执行、恢复、排障”完整工作流。

### 方向二：工具编排与安全升级

增加工具调度层、工具分类、文件 diff 审批、shell 规则和动态工具信任治理。

### 方向三：记忆与子代理升级

引入分层记忆、query-aware 检索、后台记忆提炼、子代理状态与预算治理。

### 方向四：产品化补齐

增加统一状态、事件、指标、诊断入口和多平台发布链路。

## 推荐优先级

### P0

- 阶段 0 全部
- `RFC-0002` 中的 `sessions` / `resume` / 增强 `status`

### P1

- `RFC-0002` 中的 plan mode
- `RFC-0003` 中的读写分级、diff 审批、工具批执行

### P2

- `RFC-0004` 中的分层记忆、子代理任务面板
- `RFC-0005` 中的可观测性基础设施

## 设计约束

- 不引入和现有 Trait-first 架构冲突的重型耦合层
- 第一轮优先打通 workflow 和状态模型，而不是完整 TUI
- 更严格的安全默认值必须提供清晰反馈，避免静默失败
- 新能力优先通过 `nanobot-config` 暴露，并保证 CLI / gateway 共用

## 验收标准

- 用户可以通过明确命令完成规划、执行、恢复、查看状态四类核心工作流
- 至少一类写操作具备 diff 审批或显式确认机制
- 至少一类子代理任务可后台运行并被查询状态
- `status` / `doctor` 能覆盖常见配置与运行故障
- 发布产物、文档和二进制命名保持一致
