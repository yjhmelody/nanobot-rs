# RFC-0003: 工具编排与安全机制升级

## 元信息

| 字段 | 值 |
| --- | --- |
| 文档状态 | `Proposed` |
| 实施状态 | `Not Started` |
| 提出顺序 | `3` |
| 优先级 | `P1` |
| 依赖 | `RFC-0001` |
| 最后更新 | `2026-04-01` |

## 摘要

在现有 `Tool` trait 和工具注册体系之上，增加调度、审批和权限策略层，让 `nanobot-rs` 的工具系统从“能执行”升级为“可调度、可审阅、可限制、可解释”。

## 当前不足

### 工具执行策略过于简单

- 多工具计划只执行第一个工具
- 缺少工具级依赖与批执行能力
- 没有显式区分只读工具和会变更外部状态的工具

### 写操作没有审阅环节

- `write_file`、`edit_file` 直接落盘
- shell 命令以直接执行为主
- 用户难以在关键时刻拦截高风险变更

### 权限策略能力不足

- 缺少按命令前缀、路径范围、工具类别保存规则的机制
- `restrict_to_workspace` 默认值偏宽松
- MCP 工具与自定义工具缺少统一的信任分级

## 目标

- 在不破坏现有 `Tool` trait 的前提下增加调度与审批层
- 让常见只读操作更快，让写操作更可控
- 把安全做成可配置能力，而不是散落在单个工具里的条件分支

## 方案

## 1. 为工具引入执行分类

建议为所有工具声明执行属性：

- `ReadOnly`
- `WorkspaceWrite`
- `ExternalSideEffect`
- `Network`
- `LongRunning`

用途：

- 调度器按类别决定是否并发
- 审批器按类别决定是否需要确认
- `status` / `doctor` / 日志可以展示更清晰的风险等级

## 2. 升级工具调度器

新增独立 `ToolOrchestrator`：

- 接收模型返回的多个工具调用
- 对只读工具并发执行
- 对写操作串行执行
- 在必要时插入审批步骤
- 汇总结构化执行结果后回填给 agent

### 建议执行模型

- 同一批次中的 `ReadOnly` 工具可并行
- 一旦出现 `WorkspaceWrite` 或 `ExternalSideEffect`，默认切换为串行审批执行
- 对失败工具保留局部结果，不因为单点失败吞掉整批上下文

## 3. 文件写入审批

为 `write_file` / `edit_file` 增加中间态：

- 先生成 `PendingFileChange`
- 计算 diff
- 进入审批流程
- 用户批准后再提交写入

最小版本：

- CLI 下用文本 diff 预览
- 提供 `approve once` / `approve and remember rule` / `reject`

## 4. Shell 审批与规则

把 shell 从“单次风控”升级为“规则驱动”：

- 命令分类：只读、工作区写、本机敏感、网络相关
- 支持按命令前缀保存用户规则
- 对高风险命令始终要求确认

建议新增：

- `PermissionRule`
- `ToolApprovalRequest`
- `ApprovalDecision`

## 5. 信任模式

建议定义更清晰的运行模式：

- `default`
- `plan`
- `auto-edit`
- `bypass`（仅开发/测试使用）

不同模式对工具类别施加不同默认策略。

## 6. MCP 与动态工具治理

- 首次发现新 MCP server 时记录来源和信任状态
- 未批准前只显示，不自动执行
- `status` / `doctor` 可查看动态工具来源、能力和风险分类

## 实施阶段

| 阶段 | 范围 | 状态 | 说明 |
| --- | --- | --- | --- |
| 阶段 1 | 修正工具 schema 漂移、支持多工具顺序执行、补充读写元数据 | `Planned` | 先修根本限制 |
| 阶段 2 | 引入 `ToolOrchestrator`、实现并发读 / 串行写、文件 diff 审批 | `Planned` | 建立最小安全闭环 |
| 阶段 3 | shell 规则持久化、信任模式、MCP 审批与治理 | `Planned` | 从单点风控升级为体系化治理 |

## 风险与取舍

- 调度器增加后，模型输出错误链路会更复杂，需要更好的可观测性
- 审批过多会影响体验，因此必须支持规则记忆
- 不建议一开始就实现完整 patch staging，先做最小审批闭环

## 验收标准

- agent 能在一个回合内执行多个工具，而不是只执行第一个
- 至少文件写入具备 diff 审批能力
- shell 和动态工具具备可解释的权限决策
- 用户可以看到当前工具为何被允许、阻止或要求审批
