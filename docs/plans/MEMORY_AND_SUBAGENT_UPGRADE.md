# 记忆系统与子代理升级设计

## 背景

`nanobot-rs` 已经有 session、memory、spawn 等基础设施，但它们更接近“能力存在”，还没有形成稳定的高层工作模式。`claude-code` 在这方面的启发是：把记忆做成持续背景能力，把子代理做成可观察、可恢复、可约束的工作对象。

## 当前不足

### 记忆系统过于粗粒度

- 长期记忆读取没有结合用户 query
- 写入方式偏覆盖式，难以积累结构化经验
- 没有区分 session 临时记忆、项目长期记忆、用户手工记忆

### 会话智能不足

- consolidation 有基础能力，但策略暴露不足
- 缺少“何时该压缩、何时该提炼、何时该写入长期记忆”的更高层策略

### 子代理仍偏底层执行

- 子代理创建后缺少统一状态查询
- 缺少预算、超时、工具边界、结果归并机制
- 后台工作项没有统一列表和恢复能力

## 对标亮点

适合借鉴的能力包括：

- session memory 的后台提炼
- durable project memory
- 受限工具集的后台辅助 agent
- 可展示的后台任务和可恢复的结果

## 设计目标

- 在不引入复杂外部存储的前提下，把现有本地优先能力升级成更聪明的记忆层
- 让子代理既能被利用，又不会失控
- 降低长上下文任务的 token 压力

## 方案设计

## 1. 分层记忆模型

建议拆为三层：

### Session Memory

- 与当前会话强相关
- 生命周期短
- 主要用于多轮连续任务

### Project Memory

- 面向当前 workspace
- 存储稳定事实、约定、常见命令、架构知识
- 由后台提炼流程维护

### Manual Memory

- 用户显式写入或确认的重要规则
- 优先级高于自动提炼内容

## 2. Query-aware 检索

当前 `get_context` 不利用 query，需要升级为：

- 根据 query 选择相关记忆块
- 支持按标签、来源、最近使用时间、重要度排序
- 控制回填 token 上限

最小实现不必上向量库，可以先做：

- 关键词索引
- tag + metadata 检索
- 简单评分排序

## 3. 背景记忆提炼 Worker

新增受限背景任务：

- 输入：最近对话、工具结果、用户明确反馈
- 输出：候选 session summary、候选 project memory 更新
- 约束：只允许使用少量只读工具，不允许执行写文件和危险 shell

建议设计：

- `MemoryWorker`
- `MemoryExtractionJob`
- `MemoryCandidate`

## 4. 会话压缩与记忆写入联动

将 consolidation 从“消息达到阈值就压缩”升级为更明确策略：

- 触发条件
- 压缩级别
- 是否抽取长期记忆
- 是否保留关键工具结果

建议把策略做成配置项，而不是埋在默认值里。

## 5. 子代理生命周期管理

新增统一子代理任务模型：

- `Queued`
- `Running`
- `WaitingApproval`
- `Completed`
- `Failed`
- `Cancelled`

同时记录：

- 发起 session
- 任务目标
- 可用工具范围
- 模型/预算/超时
- 结果摘要

## 6. 子代理预算与隔离

建议支持：

- 最大 token
- 最大工具调用次数
- 可用工具 allowlist
- 可选独立工作目录或工作区子目录

这能显著降低子代理跑偏或无限扩散的风险。

## 7. 结果归集

子代理完成后，不应只发一条文本消息，而应产出结构化结果：

- 摘要
- 关键发现
- 工具执行统计
- 产出文件列表
- 是否需要人工审阅

CLI 和后续 `tasks show` 可以直接消费这一结果。

## 分阶段实施

## 第一阶段

- 把记忆配置项显式化
- 实现 query-aware 检索最小版本
- 子代理状态查询最小版本

## 第二阶段

- 背景记忆提炼 worker
- project memory / manual memory 分层
- 子代理预算和 allowlist

## 第三阶段

- 会话恢复时主动注入相关记忆
- 子代理结果结构化归档
- 任务面板与 resume 深度联动

## 风险与取舍

- 记忆抽取如果做得太激进，容易写入噪声
- 子代理如果没有强预算和工具边界，排障会非常困难
- 不建议一开始引入重型向量数据库，先把本地文件和轻量索引做好

## 验收标准

- 记忆检索会根据 query 返回相关内容，而不是全量硬塞
- 至少一种长期记忆可以自动提炼并被后续会话使用
- 子代理可被列出、查询状态、查看结果摘要
- 子代理预算和工具边界可配置
